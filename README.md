# cvd-risk-prediction-mlops

<!-- Build status badge: вставьте сюда badge GitHub Actions после публикации репозитория -->

Репозиторий магистерского ML/MLOps-проекта по теме прогнозирования риска сердечно-сосудистых осложнений на основе ретроспективных данных пациентов. Проект оформлен как инженерно аккуратная база для работы с данными, обучения моделей, воспроизводимого запуска и автоматических проверок качества кода.

## Предметная область

Проект относится к области медицинских интеллектуальных систем и предиктивной аналитики. Основная задача состоит в анализе обезличенных ретроспективных клинических данных, выделении значимых факторов риска и построении моделей, помогающих оценивать вероятность сердечно-сосудистых осложнений.

## Цель проекта

Цель проекта — разработать воспроизводимый прототип интеллектуальной системы, которая:

- работает с ретроспективными медицинскими данными;
- поддерживает подготовку и версионирование датасетов;
- позволяет обучать и проверять ML-модели;
- дает основу для дальнейшего развития в сторону инференса, API и клинического использования.

## Работа с DVC

`DVC` нужен для того, чтобы не хранить сырые датасеты напрямую в Git, но при этом сохранять воспроизводимость экспериментов.

### Инициализация DVC

```bash
git init
pip install dvc
dvc init
```

### Добавление датасета

Пример для файла `data/raw/dataset.csv`:

```bash
dvc add data/raw/dataset.csv
git add data/raw/dataset.csv.dvc .gitignore .dvc/config .dvcignore
git commit -m "data: track dataset with dvc"
```

Что важно:

- реальные файлы данных не коммитятся в Git;
- в Git попадают `.dvc`-файлы и конфигурация DVC;
- сами данные хранятся локально или в remote storage.

### Настройка remote storage

Пример для `S3`:

```bash
dvc remote add -d storage s3://your-bucket-name/cvd-risk-prediction
dvc remote modify storage region your-region
```

Пример для `Google Drive`:

```bash
dvc remote add -d storage gdrive://your-google-drive-folder-id
```

Замените шаблонные значения на свои. Секреты и токены не должны храниться в репозитории.

### Получение данных на новой машине

```bash
git clone <repo-url>
cd cvd-risk-prediction-mlops
pip install -r requirements.txt
dvc pull
```

## MLflow

Для ЛР4 проект подготовлен к использованию `MLflow` как локального инструмента для управления жизненным циклом экспериментов. На текущем этапе выполнена базовая инициализация:

- `MLflow` добавлен в зависимости проекта;
- подготовлены локальные каталоги `mlruns/` и `mlartifacts/`;
- в модуле [src/utils/mlflow_config.py](C:/Projects/VKR/src/utils/mlflow_config.py) вынесены базовые пути для локального tracking backend и artifact storage.

`MLflow` в этом проекте нужен для того, чтобы централизованно хранить информацию об экспериментах по моделям прогнозирования риска сердечно-сосудистых осложнений: версии запусков, параметры, метрики и артефакты моделей. На данном шаге лабораторной подготовлена именно инфраструктура запуска локального Tracking Server и UI.

### Запуск MLflow UI локально

После установки зависимостей можно запустить локальный сервер так:

```bash
mlflow server --backend-store-uri sqlite:///./mlflow.db --default-artifact-root file:./mlartifacts --host 127.0.0.1 --port 5000
```

Если нужен только интерфейс просмотра локального каталога без отдельного server-режима, можно использовать:

```bash
mlflow ui --backend-store-uri sqlite:///./mlflow.db --host 127.0.0.1 --port 5000
```

### Где смотреть эксперименты

После запуска интерфейс доступен в браузере по адресу:

```text
http://127.0.0.1:5000
```

Именно там будут отображаться эксперименты и запуски после того, как в следующих пунктах лабораторной будет добавлено логирование параметров, метрик и артефактов.

### Запуск эксперимента

Для демонстрационного запуска эксперимента добавлен скрипт [src/models/train_experiment.py](C:/Projects/VKR/src/models/train_experiment.py). Он обучает модель `RandomForestClassifier` для задачи прогнозирования риска сердечно-сосудистых осложнений и выполняет запуск внутри `mlflow.start_run()`.

Пример запуска:

```bash
python -m src.models.train_experiment --max-depth 5 --n-estimators 200
```

Если в `data/processed/` еще нет реального подготовленного датасета, скрипт автоматически использует синтетический демонстрационный набор с медицински осмысленными признаками. Это позволяет показать работу MLflow в учебном проекте без использования реальных персональных данных.

### Какие параметры и метрики логируются

В MLflow логируются основные гиперпараметры модели:

- `max_depth`
- `n_estimators`

Также сохраняются метрики качества по согласованной схеме оценивания через стратифицированную кросс-валидацию:

- `ROC-AUC`
- `Precision`
- `Recall`

### Какие артефакты сохраняются

В MLflow сохраняются:

- изображение матрицы ошибок;
- сериализованная модель в формате `.pkl`.

Эти артефакты позволяют не только сравнивать численные результаты запусков, но и хранить итоговую модель и визуальные материалы эксперимента в одном месте.

Дополнительно для последующей регистрации в `Model Registry` модель сохраняется и в формате `MLflow Model`.

### Как запустить серию экспериментов

Для сравнения нескольких запусков добавлен сценарий [src/models/run_experiment_series.py](C:/Projects/VKR/src/models/run_experiment_series.py). Он последовательно запускает несколько `run` с разными значениями `max_depth` и `n_estimators`, но с единым набором логируемых метрик и одинаковой схемой оценки.

Пример запуска:

```bash
python -m src.models.run_experiment_series
```

После выполнения этого сценария в MLflow UI появляется несколько сопоставимых запусков одной и той же задачи прогнозирования риска.

### Как сравниваются run в MLflow UI

Сравнение выполняется внутри интерфейса `MLflow UI` в одном эксперименте. Для каждого запуска логируются одинаковые:

- гиперпараметры модели;
- метрики `ROC-AUC`, `Precision`, `Recall`;
- артефакты эксперимента.

Благодаря этому запуски можно напрямую сопоставлять по таблице метрик и параметров в интерфейсе MLflow.

### Как определяется лучшая модель

В проекте лучшая модель выбирается по метрике `ROC-AUC` как основной целевой метрике. Дополнительно при сравнении учитываются `Precision` и `Recall`, чтобы результат не был формально лучшим только по одной численной характеристике.

Для удобства добавлен вспомогательный сценарий [src/models/select_best_run.py](C:/Projects/VKR/src/models/select_best_run.py), который выбирает лучший запуск по следующему порядку:

1. максимальный `ROC-AUC`;
2. при равенстве — больший `Precision`;
3. при дальнейшем равенстве — больший `Recall`.

Пример запуска:

```bash
python -m src.models.select_best_run
```

### Почему для медицинской задачи выбрана ROC-AUC

Для задачи прогнозирования сердечно-сосудистых осложнений `ROC-AUC` удобна как основная метрика, потому что она показывает способность модели разделять пациентов с повышенным и пониженным риском независимо от конкретного порога классификации. Это особенно полезно на этапе сравнения моделей. При этом `Precision` и `Recall` остаются важными дополнительными метриками, так как для медицинской задачи имеет значение не только общая разделяющая способность модели, но и баланс между ложными тревогами и пропуском пациентов из группы риска.

### Регистрация лучшей модели в Model Registry

После выбора лучшего запуска модель можно зарегистрировать в `MLflow Model Registry` с помощью сценария [src/models/register_best_model.py](C:/Projects/VKR/src/models/register_best_model.py).

Пример запуска:

```bash
python -m src.models.register_best_model
```

В проекте используется осмысленное имя зарегистрированной модели:

```text
cvd-risk-prediction-random-forest
```

Сценарий выполняет следующие действия:

1. находит лучший `run` по `ROC-AUC` с дополнительным учетом `Precision` и `Recall`;
2. регистрирует модель из выбранного запуска в `Model Registry`;
3. переводит созданную версию модели в стадию `Staging`.

### Как модель попадает в Registry

Модель попадает в реестр не напрямую из локального `.pkl` файла, а из конкретного `MLflow run`, в котором уже сохранена модель в формате `MLflow Model`, а также зафиксированы параметры, метрики и артефакты эксперимента. Это важно, потому что Registry в жизненном цикле модели связывает версию модели с ее экспериментальным происхождением, а не просто с фактом сохранения файла на диске.

### Что означает стадия Staging

Стадия `Staging` означает, что модель выбрана как перспективная версия и подготовлена к следующему этапу проверки или интеграции. Это не финальная продуктивная версия, а управляемое промежуточное состояние жизненного цикла модели.

### Зачем нужен Model Registry

`Model Registry` нужен для того, чтобы:

- хранить версии моделей как отдельные сущности;
- связывать модель с конкретными экспериментами и результатами;
- управлять стадиями жизненного цикла модели;
- подготавливать основу для дальнейших шагов, таких как продвижение версии модели или сравнение нескольких кандидатов.

### Имитация A/B тестирования и сценарий Champion/Challenger

Для лабораторной работы добавлен отдельный сценарий [src/models/champion_challenger.py](C:/Projects/VKR/src/models/champion_challenger.py), который имитирует безопасную проверку новой версии модели по логике `Champion/Challenger`.

В этом процессе:

- `Champion` — текущая основная модель в стадии `Production`;
- `Challenger` — новая кандидатная модель в стадии `Staging`;
- решение о полном переводе в `Production` принимается только после сравнения метрик.

Пример запуска:

```bash
python -m src.models.champion_challenger
```

### Как MLflow используется в этом процессе

В рамках этого сценария `MLflow` используется для:

- хранения версий моделей в `Model Registry`;
- работы со стадиями `Staging` и `Production`;
- чтения метрик, связанных с run каждой версии модели;
- фиксации результата сравнения как отдельного evaluation run.

Сценарий сравнивает метрики `Champion` и `Challenger` и рассчитывает прирост по:

- `ROC-AUC`
- `Precision`
- `Recall`

Если новая версия показывает достаточное улучшение, она может быть автоматически переведена из `Staging` в `Production`. Если преимущество не подтверждено, модель остаётся в `Staging` и не заменяет текущую основную версию.

### Как новая модель проходит проверку

Новая модель сначала проходит этап экспериментов и регистрации, после чего попадает в `Staging`. На этой стадии она рассматривается как кандидатная версия и сравнивается с текущей `Champion`-моделью. В учебной реализации сравнение выполняется не через полноценный онлайн-трафик, а через инженерно корректную имитацию на основе уже зафиксированных метрик и версий моделей в MLflow.

### Когда модель можно переводить в Production

Перевод из `Staging` в `Production` допустим, если кандидатная модель показывает улучшение по основной метрике `ROC-AUC`, а также не ухудшает дополнительные показатели `Precision` и `Recall`. Такой подход снижает риск необоснованной замены основной модели.

### Почему этот сценарий важен для медицинского ML-проекта

Для медицинской задачи прогнозирования риска сердечно-сосудистых осложнений особенно важно избегать резкой замены модели без дополнительной проверки. Даже если новая версия выглядит лучше по одной метрике, необходимо убедиться, что она действительно безопаснее и стабильнее с точки зрения практического использования. Логика `Champion/Challenger` позволяет внедрять новые версии модели постепенно и контролируемо, сохраняя прослеживаемость решений через MLflow.
